# Domain Design Guide

For each context document purpose, owned language, capabilities, decisions,
owned data, upstream/downstream relationships, integration style, and source
IDs.

For each aggregate document root, entities, value objects, invariants,
commands, emitted events, repository needs, consistency boundary, and covered
`FR-*`/`BR-*`.

Use context relationship patterns: partnership, customer/supplier, shared
kernel, conformist, ACL, open host service, published language, or separate
ways.

Treat service boundaries as proposals evaluated by autonomy, cohesion, data
ownership, scaling, failure isolation, team ownership, and operational cost.
